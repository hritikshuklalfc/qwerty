# IMPORT_FEATURE.md — Agent Import Design Document

## Overview

The Agent Import feature allows users to import agents/swarms from **n8n workflow URLs** or **GitHub repositories** into Synapse OS. Once imported, agents are observed and controlled through the existing simulator/telemetry pipeline (SynapseContext, Socket.IO events, LiveSwarmView) — making them indistinguishable from hand-built agents at the dashboard level.

This document covers the three key design decisions required before implementation.

---

## 1. Data Model Integration

### How imported agents map to existing schemas

Imported agents use the **exact same data path** as existing agents:

| Concept | Existing System | Imported Agents |
|---------|----------------|-----------------|
| `tenant_id` | Authenticated user session | Same — scoped to user via session |
| `swarm_id` | Set by scenario runner | Generated: `import-{n8n|gh}-{uuid8}` |
| `agent_name` | Hardcoded (inventory, procurement, etc.) | Derived from workflow nodes / scan results |
| Telemetry | Socket.IO events (`agent:update`, `live:activity`, etc.) | Same — emitted by importer services |
| Dashboard rendering | `agentStatuses` from `SynapseContext` | Same — flows through identical state |

**Key design choice:** The `swarm_id` generated by an import job is just a normal string. The rest of the system treats it identically to any other swarm. No parallel data model exists.

### In-memory job tracking

Import jobs are tracked in an **in-memory store** (`importJobStore.js`) with this schema:

```
id: UUID
source_type: 'n8n' | 'github'
source_url: string
swarm_id: string         → ties into existing pipeline
status: pending | instrumenting | running | failed | stopped
detected_summary: object → preview data from scan
error_message: string?
created_at: timestamp
_process: ChildProcess?  → for kill switch
_tempDir: string?        → for cleanup
```

This store is **not persisted to Supabase** in this iteration. Jobs are lost on server restart by design — they can be re-imported.

## 2. Unified Output Format

### Both import paths produce the same output

Regardless of source (n8n or GitHub), an import job ultimately emits the same Socket.IO events that the existing pipeline already consumes:

```
┌─────────────────┐    ┌──────────────┐    ┌─────────────────────┐
│  n8n Workflow    │───▶│  Importer    │───▶│  Socket.IO Events   │
│  or GitHub Repo  │    │  Service     │    │  (agent:update,     │
│                  │    │              │    │   live:activity,    │
│                  │    │              │    │   live:demo-step,   │
│                  │    │              │    │   live:telemetry)   │
└─────────────────┘    └──────────────┘    └────────┬────────────┘
                                                     │
                                           ┌─────────▼─────────┐
                                           │  SynapseContext    │
                                           │  (agentStatuses,   │
                                           │   activityLog,     │
                                           │   pendingApprovals)│
                                           └─────────┬─────────┘
                                                     │
                                           ┌─────────▼─────────┐
                                           │  Existing Views    │
                                           │  (LiveSwarmView,   │
                                           │   Observatory,     │
                                           │   CommandCenter)   │
                                           └───────────────────┘
```

**The importer's job is**: URL → parsed structure → instrumented execution → standard Socket.IO events.

**There is no second telemetry format.** Imported agents use `agent:update`, `live:activity`, `live:demo-step`, `live:agent-handoff`, `live:telemetry`, `live:scenario-complete`, and `live:approval-added` — the exact same events the existing `SynapseContext` already listens for.

## 3. Execution / Sandbox Boundary

### Current implementation: In-process simulation

In this iteration, imported agents are **simulated** within the backend Express process using `setTimeout`-based step execution. This is acceptable because:

- n8n imports: The importer parses the workflow graph and simulates execution step-by-step, emitting realistic telemetry. In production, it would call the n8n webhook/execution API.
- GitHub imports: The importer scans the repo statically and simulates instrumented execution. No cloned code is actually executed.

### Production requirement: Sandboxed execution

For production use, **imported code must run in an isolated sandbox**:

- `child_process.spawn()` for Node.js agents (with limited permissions)
- Docker containers for arbitrary repos (one container per import job, torn down on stop/completion)

### GitHub import restriction (SECURITY)

**Open-ended GitHub repo import is restricted to a curated allowlist** in this build. This is a deliberate security decision:

> Running arbitrary cloned code from a pasted URL without Docker container isolation is a real security hole. Until Docker-by-default execution is wired for the GitHub path, imports are limited to pre-tested repos.

The allowlist includes:
- LangChain (Python)
- LangChain.js (TypeScript)
- CrewAI (Python)
- OpenAI Node SDK
- Anthropic Python SDK

**To enable unrestricted GitHub imports**, the following must be implemented:
1. Docker container execution as the default (not opt-in)
2. Network isolation (no access to host network from imported containers)
3. Resource limits (CPU, memory, execution time caps)
4. Filesystem isolation (read-only mount of the cloned repo)

This is noted here rather than silently left as a gap.

## 4. n8n Integration Details

### Health check

The n8n importer checks if the n8n Docker instance (expected at `localhost:5678`) is reachable before allowing imports. If unreachable, a clear warning is shown in the UI — not a silent failure.

### Node type mapping

n8n nodes are classified into Synapse span types:

| n8n Node Type | Synapse Span Type | Is Agent? | Risk |
|--------------|-------------------|-----------|------|
| LangChain/OpenAI nodes | `llm_call` | ✅ | Low |
| HTTP Request | `tool_call` | ❌ | Medium |
| Stripe/Shopify/Payment | `tool_call` | ❌ | **High** |
| Set/IF/Switch/Merge | `internal_step` | ❌ | Low |
| Webhook/Trigger | `trigger` | ❌ | Low |

High-risk nodes automatically generate approval requests in the existing Approval Queue.

### Connection parsing

n8n's `connections` object is parsed into handoff edges that map directly to the existing `live:agent-handoff` events, so the Live Swarm View renders them without changes.
